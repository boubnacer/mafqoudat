# Building Mafqoudat Mobile (EAS)

This covers the exact steps for a first internal Android preview build, plus what
changed in this pass and what's still a manual, one-person-with-credentials step.

## What changed in this pass

- `app.config.js` is now the **only** config file (`app.json` didn't actually exist
  in this repo despite appearances - nothing to merge there). It's the single source
  of truth for name, bundle id, scheme, permissions, icons, and the `mafqoudat://auth/callback`
  deep-link intent filter.
- `android/` (previously a committed, hand-editable native project) is now **gitignored**
  and generated fresh by Continuous Native Generation (`expo prebuild`, which EAS Build
  runs automatically) on every build. Never hand-edit files under `android/` or `ios/` -
  anything you need to change belongs in `app.config.js` or a config plugin, or it will
  be silently discarded the next time the native project regenerates.
- `src/config/api.js` no longer has hardcoded fallback URLs/client IDs. Every
  `EXPO_PUBLIC_*` var is now required - see `.env.example` for the full list and what
  each one is for. A misconfigured build now fails loudly (network errors, OAuth
  errors) instead of silently talking to whatever was baked into source last.
- `eas.json` was hand-authored (see "Why eas.json wasn't generated by the CLI" below)
  with three profiles: `development`, `preview`, `production`. All three currently
  point at the production API/Google client - there's no separate staging backend today.
- Generated placeholder `assets/adaptive-icon.png` (1024x1024) and `assets/favicon.png`
  (48x48) from the existing `assets/icon.png`, since `app.config.js` referenced both
  and neither file existed. **These are load-bearing placeholders, not real assets -
  see "Flagged for design" below.**
- Added `expo-dev-client` as a dependency (required for the `development` build profile).

## One-time setup (you need to do this - requires your Expo account)

### Why eas.json wasn't generated by the CLI

`eas build:configure` and `eas init` both require an authenticated Expo account.
This environment has no login session and I can't complete an interactive `eas login`
on your behalf, so I hand-wrote `eas.json` to match what that command would have
produced. You still need to do the account-linking part yourself:

1. Install/confirm the CLI and log in:
   ```
   cd mobile
   npx eas-cli login
   ```
2. Link this project to a real EAS project (creates a project ID under your account):
   ```
   npx eas-cli init
   ```
   This will offer to update `app.config.js`'s `extra.eas.projectId` automatically -
   **let it**, or if it can't edit the `.js` file automatically, it'll print the ID
   to paste in yourself. Either way, replace the literal string
   `"REPLACE_WITH_EAS_PROJECT_ID"` in `app.config.js` with the real value before
   building - every `eas build`/`eas submit` will fail until this is a real ID.
3. (Optional, only if you want to double check my hand-authored `eas.json` against
   what the CLI itself would generate) `npx eas-cli build:configure` - it will detect
   `eas.json` already exists and ask before overwriting anything.

### Local dev environment

```
cd mobile
cp .env.example .env
```
Fill in `.env` with real values (the committed `.env.example` defaults match what's
already in `eas.json`'s per-profile `env` blocks, i.e. the current production values -
copy them across if you don't have different ones).

## Flagged for design (placeholder-quality, do not ship as final)

- **`assets/adaptive-icon.png`** and **`assets/favicon.png`** - programmatically
  generated from `assets/icon.png` (centered, scaled to Android's adaptive-icon safe
  zone; center-cropped square for the favicon). They will look acceptable but generic -
  get real ones from design.
- **`assets/icon.png` itself is 1808x2048, not square.** Expo/EAS will scale it to fit
  wherever a square icon is needed (app icon, store listing), which can crop or distort
  it unpredictably depending on platform. This was true before this pass too; flagging
  it now because it becomes visible once `adaptive-icon.png` exists. Get a proper
  1024x1024 square icon.png before a real store submission.

## Building

All commands run from `mobile/`.

**Development build** (installs a dev client you then load JS into via `expo start`,
for day-to-day native-module development):
```
npx eas-cli build --profile development --platform android
```

**Preview build** (internal-distribution installable APK - this is the one the
acceptance criteria asks for):
```
npx eas-cli build --profile preview --platform android
```
When it finishes, EAS prints a download link/QR code for the `.apk`. Install it on a
real Android device (`adb install path-to.apk` if downloaded locally, or scan the QR
code and let Android install it directly - allow "install from unknown sources" if asked).

**Production build** (AAB, for Play Store submission - not needed for this task, listed
for completeness):
```
npx eas-cli build --profile production --platform android
```

## On-device test checklist for the preview APK

Test all of these against production (the API URL baked into the `preview` profile) -
there is no staging environment to fall back to if something's wrong here.

- [ ] **Password login** - email/phone + password against production `/auth`.
- [ ] **Google login, including the deep-link return** - see the dedicated section
      below, this is the one that behaves genuinely differently than Expo Go.
- [ ] **Posts list** - loads, pull-to-refresh works, filters/search work, offline
      banner appears in airplane mode and clears when reconnected.
- [ ] **Post creation** - full flow including an image upload, ends on the new post's
      detail screen.

### Why Google login needs special attention on a standalone build

In Expo Go, native Google sign-in rides on Expo's own auth proxy and Expo Go's own
registered URL scheme - none of that exists once the app is a standalone installed
package. Two different things change simultaneously in a preview/production build:

1. **The app now owns the `mafqoudat://` scheme itself** (via the intent filter in
   `app.config.js` → generated into the native manifest). Expo Go never registered
   this scheme, which is exactly why `src/config/api.js` notes the legacy
   browser-redirect fallback flow "only works in a standalone/dev-client build."
2. **The native ID-token flow (`USE_NATIVE_GOOGLE_AUTH=true`, the default) needs a
   Google OAuth Android client ID that's registered against this build's actual
   package name (`com.mafqoudat.app`) and signing certificate SHA-1** - EAS builds
   use an EAS-managed keystore that is **not** the same certificate Expo Go signs
   with. If the Android OAuth client in Google Cloud Console was only ever configured
   for Expo Go's shared certificate, native sign-in will fail on the preview build
   with an opaque Google-side error even though it worked fine in Expo Go.

**To verify:**

1. Get the SHA-1 EAS actually signed this build with:
   ```
   npx eas-cli credentials
   ```
   (select Android → the credential used by the `preview` profile). Confirm this
   SHA-1 + `com.mafqoudat.app` is registered as an Android OAuth client in Google
   Cloud Console, and that `EXPO_PUBLIC_GOOGLE_ANDROID_CLIENT_ID` in `eas.json` is
   that client's ID.
2. On the installed APK, tap "Continue with Google." With the native flow, this should
   open the native Google account chooser (not a browser tab) and return straight to
   the app - no visible deep-link bounce, because the native ID-token flow doesn't use
   `mafqoudat://auth/callback` at all.
3. **To specifically exercise the deep-link path** (the thing that's genuinely
   impossible to test in Expo Go): temporarily rebuild with
   `EXPO_PUBLIC_USE_NATIVE_GOOGLE_AUTH=false` (edit the `preview` profile's `env` block
   in `eas.json`, or override with `eas build --profile preview -e EXPO_PUBLIC_USE_NATIVE_GOOGLE_AUTH=false`)
   to force the legacy flow, then sign in again. This opens an in-app browser tab for
   Google's consent screen; when it redirects to the server's `mobile-callback.html`,
   confirm Android hands control back to *this installed app* (not a browser, not
   Expo Go) at `mafqoudat://auth/callback?token=...`, and that the login completes.
   Revert the env var afterward - native is the intended default.
